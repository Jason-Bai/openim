# 1. 核心定义

## 1.1 OpenClaw

OpenClaw 在本项目中定义为：

> **员工账号人格 / 身份对应的员工助手 BOT。**

OpenClaw 不是本地客户端，不是浏览器页面，也不是普通用户账号本身。

## 1.2 员工系统账号

员工先在本系统注册账号，用于登录 Web 端、使用默认 BOT、创建 BOT 接入槽位、查看 OpenClaw 员工助手的接入状态。

## 1.3 默认 BOT

默认 BOT 是本系统内置的管理入口。

默认 BOT 的职责：

- 提供帮助信息
- 创建 BOT 接入槽位
- 输出连接信息
- 查看 BOT 状态
- 删除 / 断开 BOT
- 引导员工把 OpenClaw 员工助手接入本系统

默认 BOT 不等于 OpenClaw 员工助手，也不自动连接 OpenClaw。

## 1.4 BOT 接入槽位

员工通过默认 BOT 的 `/new-bot` 命令创建一个系统内 BOT 身份，即 BOT 接入槽位。

该槽位生成：

- `bot_id`
- 连接 token
- 连接状态
- 归属员工账号

OpenClaw 员工助手拿到连接信息后，通过 Plugin 接入该槽位。

## 1.5 Plugin

Plugin 是 OpenClaw 员工助手接入本项目的官方 npm package。

Plugin 方案单独成文，v2 只引用，不展开实现细节。

参考文档：

```text
docs/plugin-npm-package.md
```

# 2. 系统边界

```text
员工 Web 端
  ↑↓
OpenClaw IM 服务端
  ├─ 用户与认证
  ├─ 默认 BOT 命令系统
  ├─ BOT 管理
  ├─ BOT Gateway
  ├─ IM 消息系统
  └─ 文件系统
  ↑↓
MySQL / Redis / 文件存储

OpenClaw 员工助手
  ↑↓ npm Plugin
BOT Gateway
```

边界说明：

- 员工 Web 端只负责人操作。
- 默认 BOT 是系统内置管理入口。
- OpenClaw 员工助手是外部 BOT Runtime。
- BOT Gateway 只负责 BOT / Plugin 接入，不和普通 Web 用户 WebSocket 混用。
- Plugin 是 OpenClaw 员工助手接入 BOT Gateway 的标准 SDK。

# 3. 项目初始化工作

项目初始化阶段需要先完成系统基础能力，不属于单个员工的 OpenClaw 接入流程。

初始化工作包括：

- 用户注册 / 登录能力
- 默认 BOT 服务
- 默认 BOT 会话入口
- 默认 BOT 命令框架
- `/help` 命令
- BOT 管理表
- BOT Gateway 基础服务
- Plugin 接入方案引用

初始化完成后，员工登录系统即可看到默认 BOT 会话。

# 4. P0 主流程：OpenClaw 员工助手接入与绑定

## 4.1 前置条件

- 员工已注册系统账号。
- 员工已登录 Web 端。
- 默认 BOT 会话可用。
- 默认 BOT 命令系统可用。
- 员工已经或即将通过公司工单获得 OpenClaw 员工助手。
- OpenClaw 员工助手支持使用官方 npm Plugin 接入。

## 4.2 接入流程

```text
员工注册并登录系统账号
  ↓
进入默认 BOT 会话
  ↓
输入 /new-bot
  ↓
系统创建 BOT 接入槽位，生成 bot_id
  ↓
输入 /connect
  ↓
默认 BOT 返回连接信息
  ↓
员工复制连接信息给 OpenClaw 员工助手
  ↓
OpenClaw 员工助手使用 npm Plugin 连接 BOT Gateway
  ↓
服务端校验 bot_id + token
  ↓
OpenClaw 员工助手发送首条握手消息
  ↓
系统标记 bot_id 为 connected
  ↓
系统建立绑定：员工账号 -> bot_id -> OpenClaw 员工助手
  ↓
Web 端显示 OpenClaw 员工助手已接入
```

## 4.3 `/new-bot`

职责：

- 创建一个 BOT 接入槽位。
- 生成 `bot_id`。
- 记录归属员工账号。
- 初始状态为 `pending`。

返回示例：

```text
已创建 BOT 接入槽位：
BOT_ID: bot_01HX7K9A2M4Q8R6T3ZP
状态：等待连接

下一步请输入 /connect 获取连接信息。
```

## 4.4 `/connect`

职责：

- 查询当前员工名下的 BOT 接入槽位。
- 让员工选择需要连接的 `bot_id`。
- 输出可复制的连接信息。

返回内容包括：

- `bot_id`
- token
- BOT Gateway 地址
- 协议版本
- npm Plugin 包名
- npm Plugin 安装命令
- 最小接入说明引用

返回示例：

```json
{
  "bot_id": "bot_01HX7K9A2M4Q8R6T3ZP",
  "token": "ocb_live_xxx",
  "gateway_url": "wss://im.company.com/bot-gateway/ws",
  "protocol_version": "bot-v1",
  "plugin": {
    "type": "npm",
    "package": "@openim/openclaw-bot-plugin",
    "version": "^1.0.0",
    "install": "npm install @openim/openclaw-bot-plugin",
    "docs": "docs/plugin-npm-package.md"
  }
}
```

## 4.5 首条握手消息

OpenClaw 员工助手连接成功后，需要通过 Plugin 发送首条握手消息。

服务端收到握手消息后，才完成绑定。

握手消息用于证明：

- OpenClaw 员工助手已真实接入。
- `bot_id + token` 可用。
- Plugin 协议版本匹配。
- 该 BOT 可以收发消息。

握手完成前，BOT 状态保持为 `pending` 或 `authenticating`。

握手完成后：

```text
bots.connect_status = connected
bots.bind_status = bound
user_bot_bindings.status = active
```

# 5. 默认 BOT 命令

第一版默认 BOT 保留 6 个命令，但需要明确命令语义。

## 5.1 `/help`

展示默认 BOT 可用命令和接入说明。

## 5.2 `/my-bots`

查看当前员工创建的 BOT 接入槽位和连接状态。

## 5.3 `/new-bot`

创建新的 BOT 接入槽位，生成 `bot_id`。

## 5.4 `/delete-bot {bot_id}`

删除当前员工名下未使用或已断开的 BOT 接入槽位。

要求：

- 只能删除归属当前员工的 BOT。
- 已连接 BOT 删除前需要先断开或二次确认。

## 5.5 `/connect`

输出 BOT 接入连接信息。

该命令不直接连接 OpenClaw 员工助手，只生成并展示连接所需信息。

## 5.6 `/disconnect`

断开当前员工名下已连接 BOT。

断开后：

- BOT Gateway 主动关闭连接。
- BOT 状态变为 `disconnected`。
- 绑定关系可保留为历史，也可以按策略标记为 `revoked`。

具体策略后续单独确认。

# 6. 模块拆分与优先级

## P0-A：项目初始化

目标：让员工能登录系统，并看到默认 BOT 管理入口。

范围：

- 用户注册
- 用户登录
- 认证 token / session
- 默认 BOT 服务
- 默认 BOT 会话入口
- 默认 BOT 命令框架
- `/help`
- 基础数据库表

验收：

- 员工可注册账号。
- 员工可登录 Web 端。
- 聊天列表出现默认 BOT。
- 输入 `/help` 有稳定返回。

## P0-B：OpenClaw 员工助手接入

目标：员工可以通过默认 BOT 创建接入槽位，并把 OpenClaw 员工助手接入系统。

范围：

- `/new-bot`
- `/connect`
- `bot_id` 生成
- token 生成与哈希存储
- BOT Gateway
- bot_id + token 鉴权
- Plugin 接入引用
- 首条握手消息
- 员工账号与 BOT 绑定

验收：

- `/new-bot` 可以创建 `pending` 状态 BOT。
- `/connect` 可以返回完整连接信息。
- OpenClaw 员工助手使用 Plugin 可以连接 BOT Gateway。
- token 错误时连接失败并有明确错误。
- 握手成功后系统建立绑定关系。
- Web 端可看到该 BOT 已连接。

## P0-C：接入可验证与可排查

目标：接入过程出现问题时可定位、可复现、可验证。

范围：

- BOT 连接状态
- `first_message_at`
- `last_seen_at`
- 接入日志
- 错误码
- trace_id / request_id
- 心跳超时
- 断线重连状态

验收：

- 可以区分 `pending`、`authenticating`、`connected`、`disconnected`、`revoked`。
- 鉴权失败能看到原因。
- 90 秒无心跳可标记离线。
- 断线重连后状态恢复。
- 每次接入有可追踪日志。

## P1：基础 IM 能力

目标：员工和已接入 BOT 能进行基础聊天。

范围：

- 会话列表
- 单聊消息
- Web 用户 WebSocket
- 消息落库
- 历史消息
- 消息已发送 / 失败状态

验收：

- 员工可以向已接入 OpenClaw 员工助手发消息。
- OpenClaw 员工助手可以回复员工。
- 刷新页面后历史消息仍可查看。
- WebSocket 断开后可恢复。

## P2：员工自建 BOT

目标：员工后续接入自己的 BOT 时，复用 OpenClaw 员工助手已走通的接入链路。

范围：

- `bot_type`
- 自定义 BOT 名称
- 多 BOT 管理
- 自建 BOT 权限范围
- 自建 BOT 与 OpenClaw 员工助手区分

验收：

- 同一员工可拥有多个 BOT。
- 可以区分 `openclaw_assistant` 和 `custom_bot`。
- 自建 BOT 可以复用 `/new-bot`、`/connect`、Plugin、BOT Gateway。

## P3：主区、子区、文件、运维规模化

目标：完善 IM 系统能力和上线运行能力。

范围：

- 主区
- 子区
- 区成员
- 区消息
- 文件上传
- 文件下载
- 文件消息
- Redis 在线状态
- 日志
- 监控
- 告警
- 部署
- 2000 人压测

验收：

- 主区 / 子区消息正常。
- 文件上传下载正常。
- 2000 人在线压测稳定。
- 关键指标可监控。
- 服务异常有告警。

# 7. 技术栈草案

## 7.1 前端

- Web 应用
- TypeScript
- React 或 Vue 任选其一，后续需要确认
- WebSocket 客户端
- 文件上传组件

## 7.2 后端

- Java Spring Boot 或 Node.js NestJS 任选其一，后续需要确认
- REST API
- WebSocket
- BOT Gateway
- 默认 BOT 命令系统

## 7.3 数据层

- MySQL：用户、BOT、绑定、消息、区、文件元数据
- Redis：在线状态、连接状态、短期会话、限流
- 对象存储：文件内容

## 7.4 Plugin

- npm package
- TypeScript
- Node.js 18+
- WebSocket JSON 协议
- 详细方案见 `docs/plugin-npm-package.md`

# 10. 待确认问题

2. 后端技术栈选择 Spring Boot 还是 NestJS。
3. `/disconnect` 后绑定关系是保留为历史，还是立即 revoke。
